**The as-written math DOES warp.** The composite is a coherent 360 panorama: horizontal-camera tiles are bowtie-shaped (valid width tapers 938px mid → ~240px at top/bottom edges), gridlines burned into the source visibly bow (~60–70px sag across a tile), sky cameras (G/H/J) smear from 436px wide at their top to 1735px at mid — dramatic equirect stretching near zenith. Everything below lat ≈ −16° is black (all cams pitched up). Doubled/ghosted power lines arc across the sky where seams cross-fade. So "output never looks warped" is NOT literally true of this code path — but the warp is subtle where a human looks: horizontal tiles are uniformly **minified ~2.2×** (2048px source → ~930px tile at 3840 output), which a casual viewer reads as "scaled down, not re-projected," and the 200–340px-wide cross-fades erase any hard seam that would betray misalignment. In a fit-to-window preview the equirect also stretches anisotropically to the window (`StitchShaders.metal:52` `normPos = in.position.xy / drawable`), further disguising the signature.

## 1. Focal derivation — the structural poison
**Native JSON path** (the only path ever actually used): values are baked by `tools/calibrate.py` — `focal_mm_to_px(12.0, 2048)` with hardcoded `SENSOR_WIDTH_MM = 22.56` → f_px = 12·2048/22.56 = **1089.36** (hfov 86.46°), matching calibration.json. The formula matches its own assumption — but the assumption is unvalidated.

**What PTGui actually measured** (`Roll01_Clip04/03:21:24_Mercy_Ptgui/mercy01.pts`, PTGui Pro 12.24, project_v33): `globallenses[0].lens.params: focallength = 13.0222 mm, sensordiagonal = 30.56 mm` on 6144×3240 stills → sensor width 27.03mm (Komodo 6K full width) → **hfov = 2·atan(13.515/13.022) = 92.1°**. The 2048×1080 clips are an exact 3× downscale of 6144×3240 (identical aspect), so the same 92.1° applies → correct f_px ≈ **987**, not 1089. Runtime focal is **~10.4% too long**; each horizontal tile renders ~5.7° too narrow. If the 22.56mm "2K mode" sensor-width guess is also wrong (full-readout downscale would be 27.03mm → f_px 909), the uncertainty band is 909–1089 px (**20%**). No revision round could converge because intrinsics were hand-locked wrong and only extrinsics were ever tuned. Sky lens: PTGui says 9.885mm (hfov ~107.5°) vs runtime 817.02px (102.8°) — same ~5° error.

**.pts path** (`CalibrationData.swift:41-133`): formula `focalPx = width/(2·tan(hfovRad/2))` (lines 76-77) is correct, **but it can never execute on a real PTGui file**. It requires `root["imagegroups"]` — real v33 puts imagegroups under `project`; it requires per-image `hfov/yaw/pitch/roll/width/height` — real v33 stores yaw/pitch/roll in `group.position.params`, focal as mm in `project.globallenses[].lens.params` (no hfov field anywhere), size in `group.size`; it requires `project.outputsize.w/h` — this project uses `{"mode":"relative","fractionofoptimumsize":1}`. Loading `mercy01.pts` throws `PTSError.invalidFormat` at the first guard. The only ".pts" it can parse is the fake one `calibrate.py:write_ptgui_pts` (line 834) writes — a round-trip of its own numbers. **The PTGui bundle-adjusted calibration (correct focals, distortion, shifts, 60° yaw spacing) never reached the GPU. This single fact makes all PTGui-side recalibration rounds futile by construction.** (`CalibrationLibrary.swift:39` additionally prefers .json over .pts with the same stem.)

## 2. Principal point / d,e shift
`CalibrationData.swift:116`: `principalPoint: [width/2, height/2]` — forced center, always. d/e read from `img["d"]/img["e"]` (lines 95-96) — keys that exist only in the self-written fake .pts, always 0. Real PTGui v33 stores shift as `lens.shift.params.{longside: 0.012146, shortside: 0.000278}` — normalized fractions → **74.6px @6144 = 24.9px @2048** horizontal principal offset for all six horizontal cams (sky lens: 18px vertical). Metal applies d/e as `u = f·xd + cx + d` (RemapCompute.metal:71-72) in *runtime* pixels; even the classic-format d/e would have been in *calibration-resolution* (6144) pixels — an unhandled 3× scale trap. Net: a systematic ~25px source (≈11px output) mis-registration was discarded, plus PTGui distortion a=−0.0023/b=−0.0106/c=+0.0557 (lens0) zeroed — c alone displaces mid-field points ~0.9% of r (~10–30px source scale). These are exactly seam-sized errors: every seam ghosts by 10–40px no matter how yaw/pitch/roll are tuned.

## 3. Equirect loop / output assumptions
`RemapCompute.metal:26-27`: `lon = ((x+0.5)/outW)·2π − π; lat = ((y+0.5)/outH)·π − π/2` — correct full 360×180. outputSize comes from calibration.json `[3840,1920]` (2:1 ✓) via `AppDelegate` → `RemapGenerator.generate` (verified call sites lines 107-111, 396-400). No band-render bug. Display: `StitchShaders.metal:52-55` maps fragment→LUT by independent x/y normalization and **nearest-texel `remaps.read`** at drawable scale — non-2:1 windows squash the pano and small windows point-sample the LUT (blocky weights), but neither creates an identity warp.

## 4. Rotations
`CalibrationData.swift:188-214`: R = Ry·Rx·Rz built with `simd_float3x3(rows:)` (row-major as written), R_inv = transpose, uploaded once — no double-inverse. Numerically verified: y-down world, top row = sky, camera yaw +26° centers at lon +26°, pitch +10° tilts view up. Self-consistent with `calibrate.py:ypr_to_rotation_matrix` (same author, same convention). PTGui yaw sign compatibility is moot since PTGui yaws are never actually imported. **No bug here** — the sim panorama assembles correctly.

## 5. float16 LUT quantization
Measured on the real grid: UV error **max 0.50px, mean 0.17px** at 2048-wide source (half ULP near 1.0 = 2⁻¹¹). Sub-pixel shimmer at worst. **Not a factor.**

## 6. Feather widths (quantified)
Runtime yaw spacings (from calibration.json): A→B 54.67°, B→C 53.28°, C→D 54.47°, D→E 54.12°, **E→F 89.12°**, F→A 54.36°. With hfov 86.46°: typical seam overlap ≈ 31.8° ≈ **339 output px** at the equator of 3840. Weight ramp (smoothstep edges reversed, 1.0→0.7: full weight inside 70% of half-width): falloff spans source angle 33.4°→43.2° ≈ 9.9° ≈ 106 output px per camera; combined two-camera dissolve zone ~200–340px wide. Vertical (1.0→0.4): ramp 11.2°→26.4° off-axis ≈ 162 output px. So every seam is a ~300px dissolve — misalignment from wrong intrinsics becomes low-contrast double-exposure ghosting (clearly visible in `full_stitch_sim.jpg` power lines), and parameter tweaks produce changes too diffuse to judge. **Bonus defect: the E–F gap is 89.12° > hfov 86.46° → overlap −2.7° → a genuine coverage hole/black band near lon ±180°.** The 54.6° spacings trace to hardcoded seed yaws in `calibrate.py:46-54`; PTGui measured the true rig at ~60° spacing (59.6/61.2/58.8/59.5/61.1/59.7) — the runtime yaw geometry is wrong by ~5° per seam with the residual dumped between E and F.

## 7. Resampling
Source color: `sample(filter::linear)` ✓. LUT: `remaps.read` nearest per drawable pixel — no interpolation of UVs across LUT texels; at sub-3840 drawables this quantizes the warp field slightly but cannot flatten it. Nothing masks warping.

## Verdict
**Would a viewer see stretching/bowing at 3840×1920 with the as-written math? YES** — bowtie tile outlines, ~7% gridline bow in horizontal tiles, 4× horizontal smear in sky tiles, black below −16° lat (all demonstrated in the sim images). The geometry pipeline (equirect loop, ray math, rotation chain, LUT storage, sampling) is **correct and trustworthy as reference**. If Andrew's recollection is that literally nothing warped, either he judged from small/moving previews where 2.2× minification + soft dissolves read as "unprocessed," or a different render path (grid view) was on screen — but the shipped math itself warps.

**Top 3 candidate bugs for "revisions never converged":**
1. **Hand-locked wrong intrinsics** — f_px 1089.36 vs PTGui-measured ~987 (10–20% off depending on the unvalidated 22.56mm sensor guess), distortion zeroed (PTGui c=+0.056 significant), principal forced center (PTGui shift ≈25px @2K). Extrinsic-only tuning against wrong intrinsics can never align seams.
2. **The .pts importer cannot read real PTGui files** (root-vs-project imagegroups, per-image hfov that doesn't exist, relative outputsize) — the one correct calibration was structurally unreachable; only the tool's own fake .pts round-trips.
3. **Wrong yaw geometry + giant feathers hiding it** — hardcoded ~54.6° seed spacing vs true ~60° (incl. a negative-overlap 89.1° E–F hole), with ~300px smoothstep dissolves converting every misalignment into faint ghosting so each revision looked like "nothing changed."

**Day-1 validation gate for the Python rewrite:** (a) warp one camera with a burned-in grid and require visible bow/bowtie (compare against `tile_A_grid.jpg`); (b) assert Σ(adjacent yaw gaps)=360° and every gap < hfov; (c) derive f_px from the real PTGui project (`project.globallenses[].lens.params.focallength` + `sensordiagonal`, scaled to runtime resolution) and require agreement with the EXIF/lens-spec value within 2%; (d) carry PTGui a/b/c/shift through (shift is normalized longside/shortside fractions, not pixels); (e) diff a single-cam warp against PTGui's own layer render of the same frame before any blending exists.
